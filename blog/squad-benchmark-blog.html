Evaluating Squad: Multi-Agent Coordination for GitHub Copilot CLI

Same model. Same tasks. Isolating the coordination layer.

June 2026 · Tamir Dresher

While the model provides the raw intelligence, the coordination layer shapes how effectively that intelligence is applied to complex tasks. Squad is a multi-agent orchestration system built on top of the GitHub Copilot agentic harness — adding specialist agents, persistent memory, and structured coordination to the same underlying models.

In this report, we evaluate the impact of Squad's coordination layer across multiple benchmarks, holding the model constant and isolating the orchestration variable — following the same methodology as GitHub's harness evaluation approach.

Architecture

Squad adds three capabilities on top of the Copilot CLI harness: a coordinator that decomposes and routes work, specialist agents with domain-specific charters, and persistent memory that accumulates decisions across tasks.

Squad adds multi-agent coordination on top of the Copilot CLI harness. Same model powers every agent.

Controlled ablation: isolating Squad components

Holding the model constant (Claude Opus 4.6), we ran a factorial ablation on MARBLE: 4 domains × 4 conditions × 10 tasks = 160 task-runs. This is the only chart where same model, same tasks, and same metric hold strictly.

MARBLE factorial ablation — completion rate by condition (n=160, same model)

Factorial ablation: same model (Claude Opus 4.6), same tasks, same metric. n=10 per condition per domain.

Key finding: Coordination is the dominant factor. Coord-only (92.5%) vastly outperforms No-Squad (60%) — a +32.5pp lift from routing alone. Memory without coordination actually hurts performance (48.75% vs 60%), suggesting unstructured context adds noise.

Per-domain breakdown

The impact of coordination varies dramatically by domain. Database tasks show the largest gap — zero completion without Squad vs 100% with it. This is because database tasks require multi-step reasoning and schema understanding that benefits most from structured agent routing.

One harness, structured coordination

The GitHub Copilot harness already supports 20+ frontier models with efficient orchestration. Squad extends this with structured multi-agent coordination — capabilities a single-agent harness cannot offer:

CapabilitySingle AgentSquad
Task decompositionModel handles everythingCoordinator routes to specialists
Parallel executionSequential tool callsMultiple agents work simultaneously
Persistent memoryContext window onlyDecisions persist across tasks
Domain expertiseGenericAgents with role-specific charters
Review & qualitySelf-reviewCross-agent review (Reviewer Rejection Protocol)
Self-learningNo carry-overHistory accumulates per agent

Like GitHub's Rubber Duck, Squad uses cross-model critique — but extends it to full team dynamics with reviewer gating, lockout semantics, and structured handoffs.

Methodology & Limitations

Controlled experiment (ablation charts): Same model (Claude Opus 4.6), same 160 tasks, same metric (binary file output within 600s timeout). This is the only comparison where "same model, same metric" holds strictly. n=10 per condition per domain. Standard deviations computed across 4 domains.

Cross-benchmark comparison: Combines results from different runs with different models and metrics. Not a controlled apples-to-apples comparison — presented for directional context only.

  • MARBLE: "No Squad" bar = same-model ablation (controlled). Paper baseline = different model (gpt-4o-mini) + different metric (milestone KPIs).
  • DevBench: Squad uses GPT-5.4; best baseline is GPT-5.5 (#2 on leaderboard). No same-model ablation available.
  • SWE-bench Lite: Baseline estimated from public Copilot agent reports. No verified same-model-without-Squad run performed.

Metric note: MARBLE "completion rate" = agent produced an output file within timeout. This measures output reliability, not code correctness. DevBench uses pass@1 (test execution). SWE-bench uses pass@1 (patch applies + tests pass).

Variance: MARBLE ablation σ values: Full Squad σ=0 (all 100%), Coord-only σ=4.3%, No-Squad σ=35.4%, Memory-only σ=17.5%. High No-Squad σ driven by 0% database domain. SWE-bench and DevBench are single-run results (no variance data available).

Cost: All costs from actual Claude API billing (input + output tokens). Memory-only cost not recorded — estimated at ~$1.50/task based on context size.
