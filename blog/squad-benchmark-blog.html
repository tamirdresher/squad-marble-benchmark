Evaluating Squad: Multi-Agent Coordination for GitHub Copilot CLI

A controlled MARBLE ablation, plus directional benchmark comparisons of Squad's coordination layer.

June 2026 · Brady Gaster & Tamir Dresher

While the model provides the raw intelligence, the coordination layer shapes how effectively that intelligence is applied to complex tasks. Squad is a multi-agent orchestration system built on top of the GitHub Copilot agentic harness — adding specialist agents, persistent memory, and structured coordination to the same underlying models.

In this report, we evaluate the impact of Squad's coordination layer. The core result is a controlled factorial ablation on MARBLE — same model, same tasks, same metric — which is the only place where the orchestration variable is strictly isolated. We supplement it with directional comparisons on three further benchmarks (DevBench, SWE-bench Lite, TerminalBench 2.0), some of which use different models, estimated baselines, or different harness paths and are therefore contextual, not controlled. We label each comparison accordingly, following the same methodology as GitHub's harness evaluation approach.

Architecture

Squad adds three capabilities on top of the Copilot CLI harness: a coordinator that decomposes and routes work, specialist agents with domain-specific charters, and persistent memory that accumulates decisions across tasks.

Squad adds multi-agent coordination on top of the Copilot CLI harness. Same model powers every agent.

Controlled ablation: isolating Squad components

Holding the model constant (Claude Opus 4.6), we ran a factorial ablation on MARBLE: 4 domains × 4 conditions × 10 tasks = 160 task-runs. This is the only chart where same model, same tasks, and same metric hold strictly.

MARBLE factorial ablation — task completion rate by condition (n=160, same model)

Factorial ablation: same model (Claude Opus 4.6), same tasks, same metric. n=10 per condition per domain.

Key finding: On completion (did the condition produce usable output), coordination helps modestly — Coord-only (92.5%) edges out No-Squad (85%), a +7.5pp lift from routing. Memory without coordination clearly hurts (42.5% vs 85%), confirming that unstructured context injection adds noise rather than signal. (Note: this "memory-only" condition injects a Squad-generated decisions.md into a raw agent, so it measures Squad-flavored memory without coordination — not generic persistent memory.) But completion is not correctness. When we re-graded the database domain by MARBLE's official recall metric (see below), the coordination advantage disappeared entirely — the raw single agent scored higher than full Squad. Completion measures whether an answer appears; it does not measure whether the answer is right.

Completion vs. correctness: the controlled 4-domain result

Completion only asks “did an answer appear?” — it says nothing about whether the answer is right. That is the same bar GitHub’s own harness post held itself to (SWE-bench pass@1, TerminalBench verifiers grade correctness, not output), so we hold Squad to it too. Earlier we could grade correctness cleanly on only two domains, because the four conditions’ archived transcripts had drifted out of task alignment — the full-Squad set came from the 400-task production run while the others came from separate reruns, so “coding task 10” often meant different MARBLE programs across conditions. Rather than drop coding and database, we re-ran all four conditions from scratch on the identical task IDs (1, 10, 20, …, 90) in every domain — so “task N” is guaranteed to be the same MARBLE task in every condition, by construction. We then graded every one of the 80 fresh transcripts two ways, with one identical judge (Claude Opus 4.6) and one identical prompt across all conditions:

  • Milestone-KPI — MARBLE’s signature metric: the fraction of the task’s gold milestones a judge confirms the output actually achieved.
  • Output-quality rubric — a 1–5 score for correctness, completeness, and quality of the produced artifact.

The correctness finding — and a correction to our earlier one. On same-task, same-model, uniformly-judged output, coordination helps or ties in every domain. Full Squad leads overall on both metrics (81.1% KPI / 4.10 quality) — a clean +3.9pp KPI and +0.34 quality over the raw single agent, with coord-only essentially tied. Just as important, this controlled rerun overturns two earlier findings that turned out to be measurement artifacts, not real effects:

  • The coding “reversal” is gone. With aligned tasks, Full Squad coding (81.7% KPI) now leads the single agent (78.3%). The earlier “single agent wins coding” was task misalignment, not a coordination weakness.
  • The database “coordination hurts” claim is refuted. With one uniform grader on identical tasks, the coordinated conditions now match or beat the single agent (coord-only 71.7%, Full Squad 66.7% vs no-Squad 65.0%). The earlier “single agent scored higher recall” came from grading divergent tasks with inconsistent extractors — exactly the risk our reviewers flagged.
  • Memory without coordination is the weakest condition of all four (65.8% / 3.58). Injecting Squad’s decisions.md into a raw agent with no coordinator to act on it actively drags results down — memory only pays off when there is coordination to use it.

The honest bottom line: coordination’s correctness benefit is real, consistent across domains, and modest — a few points of KPI and about a third of a rubric point — not the double-digit swings the earlier misaligned data implied in either direction.

Methodology & Limitations

Controlled experiment (ablation charts): Same model (Claude Opus 4.6), same 160 tasks, same metric (binary file output within 600s timeout). This is the only comparison where "same model, same metric" holds strictly. n=10 per condition per domain. Standard deviations computed across 4 domains. The ablation achieved 40/40 = 100% for Full Squad. The broader 400-task run achieved 397/400 = 99.25% (3 timeouts in coding and database).

Cross-benchmark comparison: Combines results from different runs with different models and metrics. Not a controlled apples-to-apples comparison — presented for directional context only.

  • MARBLE: "No Squad" bar = same-model ablation (controlled). Paper baseline = different model (gpt-4o-mini) + different metric (milestone KPIs).
  • DevBench: Squad uses GPT-5.4; best baseline is GPT-5.5 (43.5%). A same-model ablation is planned.
  • SWE-bench Lite: Baseline estimated from public Copilot agent reports. No verified same-model-without-Squad run performed.
  • TerminalBench 2.0: Same model (Claude Opus 4.6), but directional only — not a controlled comparison. 20 of 89 available tasks sampled, selected by sequential order (not cherry-picked). The two conditions ran through different harness paths — Squad via a host-side orchestrator (run-terminalbench-host-task.ps1) and the baseline via the native terminal-bench harness — though both execute the identical task workloads and are scored by each task's own Docker verifier. The margin is a single task (16/20 vs 15/20) on n=20, so treat it as suggestive only. Squad can run under the native harness (verified in smoke tests); the full 89-task suite is future work, not a capability blocker.

Metric note: Metrics differ by benchmark. MARBLE (ablation charts) reports completion = agent produced an output file within a 600s timeout; the database domain was additionally re-graded for correctness using MARBLE's official recall metric (see “Completion vs. correctness”). DevBench: pass@1 (test execution). SWE-bench: pass@1 (patch applies + tests pass). TerminalBench: verifier script passes. We use "completion" where the metric only checks that output appeared, and "resolution/pass" where output is verified correct.

Correctness & quality study (all four domains): To measure correctness/quality rather than just completion, we graded transcripts with MARBLE's milestone-KPI (fraction of gold milestones achieved) plus a 1–5 output-quality rubric, using one identical judge (Claude Opus 4.6) and prompt across all four conditions. An earlier attempt hit a task-alignment problem: the archived transcripts did not always run the same MARBLE task under the same task index (full-Squad came from the 400-task production run; the others from separate ablation reruns). A rare-token anchor audit showed research (8/10) and bargaining (10/10) were cleanly aligned, but coding (1/10) and database (0/10) were not. Rather than exclude those domains, we re-ran all four conditions from scratch on the identical task IDs (1,10,20,…,90) in every domain — 80 fresh transcripts total — so “task N” is the same MARBLE task in every condition by construction. We then graded all 80 with the single Opus-4.6 judge. The full 4-domain table (research n=8 + bargaining n=10 + coding n=10 + database n=10 = 38 tasks × 4 conditions) reports the result; database KPI blends process and final-answer milestones, so we lean on its rubric for correctness. We cross-checked research/bargaining against an independent gpt-4o rubric (judges agree on ordering, differ on fine placement — deltas are directional). Raw reruns are in results/aligned_rerun/; scripts, gold milestones, and per-cell grades in quality_ablation/.

Variance: MARBLE ablation completion σ (n=10 per cell, across 4 domains): Full Squad σ=0%, Coord-only σ=4.3%, No-Squad σ=11.2%, Memory-only σ=26.8%. (An earlier version reported No-Squad σ=35.4%; that was inflated by a fabricated 0% database figure, now corrected to 100% completion after re-running all 10 tasks.) Full 400-task run: 397/400 = 99.25% (3 tasks timed out without output). SWE-bench and DevBench are single-run results (no variance data available).

Statistical power: With n=10 per cell, a two-proportion z-test can only detect effects of roughly ~25pp at p<0.05. After correcting the database data, the same-model completion lift (full Squad 100% vs no-Squad 85% = +15pp) and the database correctness differences (≤20pp between conditions) fall below this threshold — i.e., they are within noise and should be read as directional, not statistically significant. We report them honestly rather than overclaiming; larger runs are needed to establish significance.

Cost: All costs from actual Claude API billing (input + output tokens). Memory-only cost not recorded — estimated at ~$1.50/task based on context size.
